# Тестовое задание для effective mobile

## Запуск
``` 
docker compose -f docker-compose.prod.yml up --build
```

## Описание

Данный проект представляет собой бэкенд-сервис на базе Flask с использованием SQLAlchemy. Реализованы модели пользователей, jwt-токенов и разрешений ролей. Сервис поддерживает регистрацию, аутентификацию (JWT), управление пользователями и правами доступа.

## Структура базы данных

### Таблица `user`
Хранит информацию о пользователях.

| Поле        | Тип        | Описание                                      |
|-------------|------------|-----------------------------------------------|
| id          | INTEGER    | Первичный ключ, автоинкремент                 |
| name        | VARCHAR    | Имя пользователя                              |
| last_name   | VARCHAR    | Фамилия                                       |
| email       | VARCHAR    | Email (уникальный)                            |
| password    | VARCHAR    | Хэш пароля                                    |
| is_active   | BOOLEAN    | Удален ли пользователь (по умолчанию False)   |
| role        | INTEGER    | Номер роли   |

### Таблица `refresh_token`
Хранит refresh-токены для продления доступа.

| Поле        | Тип        | Описание                                      |
|-------------|------------|-----------------------------------------------|
| id          | INTEGER    | Первичный ключ, автоинкремент                 |
| user_id     | INTEGER    | Внешний ключ на `user.id`                     |
| token       | VARCHAR    | Строка токена                                 |
| expires_at  | TIMESTAMP  | Время истечения токена                        |
| revoked     | BOOLEAN    | Отозван ли токен (по умолчанию False)         |

### Таблица `role_permission`
Хранит разрешения, связанные с ролями.

| Поле        | Тип        | Описание                                      |
|-------------|------------|-----------------------------------------------|
| id          | INTEGER    | Первичный ключ, автоинкремент                 |
| name        | VARCHAR    | Название разрешения (уникальное)              |
| role_id     | INTEGER    | Идентификатор роли (внешний ключ)             |

### Связи
- `RefreshToken.user_id` → `User.id` (один пользователь может иметь много токенов)
- `RolePermission.role_id` → роль, определённая в коде (нет отдельной таблицы ролей)

## Допущения

1. **Роли не хранятся в БД**. Роли пользователей определяются через поле `role` в таблице `user`. Для проверки прав используется константа `ADMIN_ROLE = 20`. Пользователи с `role_id >= 20` считаются администраторами и могут управлять разрешениями.
2. **Аутентификация** основана на JWT, причём для генерации и проверки используется алгоритм ES256. Секретный ключ передаётся в `JWTService`.
3. **Хэширование паролей** выполняется через `werkzeug.security` (generate/check password hash).
4. **Транзакции** управляются через контекстный менеджер `DBTransaction`, который автоматически коммитит или откатывает изменения.
5. **Методы репозиториев** не содержат `db.session.commit()` – это делегируется сервисному слою с использованием `DBTransaction`.
6. **Токены**. На уровне моделей и репозиториев реализована работа с refresh-токенами, однако, на уровне сервиса бизнес-логики работа с ними была упущена в сторону скорости разработки.
